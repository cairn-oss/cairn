# Cairn v0.3.0 — breadth with the same spine

A second IaC target, shareable policy, and cost review at the pull
request — all through the unchanged v0.1 findings model. This is the
release that proves the architecture: a whole new discipline slots in
without touching the pipeline.

## Highlights

- **Kubernetes manifest scanning.** Deployments, StatefulSets, DaemonSets,
  Jobs, CronJobs and Pods flow through the same pipeline as Terraform.
  Six rules: privileged containers (K8S001), root containers (K8S002),
  hostPath mounts (K8S003), unpinned images (K8S004), missing resource
  limits (K8S005), missing probes (K8S006). Terraform and Kubernetes scan
  together, in one pass.
- **Policy packs.** `extends:` in `.cairn.yaml` layers bundled packs
  (`strict-security`, `cost-guard`) or relative in-repo packs under your
  overrides — traversal-safe by construction.
- **`cairn diff`.** Estimated monthly cost movement between two
  configurations, with an optional `budget.max_monthly_increase` gate that
  fails CI on breach — cost review before merge, not on next month's bill.
- **HTML report** (`--format html`): one self-contained file, no
  JavaScript, no external assets.
- **Open-core boundary in code.** `cairn license` states exactly what is free forever (everything in this
  repo) versus what a future commercial package adds; the core never checks
  a key, phones home, or degrades.

## Upgrade

```bash
pip install --upgrade cairn-iac   # -> 0.3.0
```

Existing configs remain valid. Repositories containing Kubernetes YAML
will now surface K8S findings — stage adoption with `disabled_rules` or
`min_severity` if desired.

## Security

Kubernetes parsing treats repositories as hostile input (2 MiB ceiling,
YAML anchor-bomb guard, no symlink traversal, per-file containment);
`extends` refuses absolute paths and directory escapes. Every path is
regression-tested.

## Compatibility & rollback

`pip install cairn-iac==0.2.0` restores prior behavior. Configs using
`extends`/`budget` fail loudly on 0.2.0 (unknown key) rather than
silently changing meaning.

## Cloud coverage note

This release is **not yet fully cloud-agnostic**. Its rule packs cover AWS
and Kubernetes (Kubernetes scanning is platform-agnostic and works
on-prem or on any cloud).
Terraform for other providers (Azure, GCP, Oracle, VMware
vSphere, ...) is *parsed* but not yet *checked*. Broader multi-cloud and
on-prem coverage arrives in later releases (Azure/GCP/vSphere packs in
v0.4.5; coverage-transparency reporting in v0.5.1, which makes unscanned
resources explicit rather than reporting a misleading "clean"). See
[docs/coverage.md](../docs/coverage.md).
