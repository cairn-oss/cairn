# Cairn v0.5.1 — coverage transparency

The honesty fix that makes Cairn's cloud-agnostic architecture
trustworthy in practice: a scan never claims "clean" for resources it
didn't actually check.

## Highlights

- **No more misleading "clean".** Previously, scanning Azure/GCP/Oracle
  Terraform with an AWS-only rule set reported "clean" — because nothing was
  wrong, but only because nothing was checked. Now Cairn says
  "checked N of M resource(s)" and lists the **not-scanned** types, clearly
  labelled a coverage gap rather than a clean result.
- **`cairn providers`** shows exactly which clouds and on-prem platforms
  are covered (AWS, Azure, GCP, Kubernetes, vSphere) and the rule count for
  each.
- **Precise gaps.** Providers without a rule pack (Oracle Cloud,
  DigitalOcean, Alibaba, IBM, OpenStack, ...) are named rather than lumped
  into "other", so the gap message tells you exactly what's unscanned.
- **JSON coverage block** (additive): `coverage.fully_covered`,
  `uncovered_types`, `uncovered_by_provider`, and
  `summary.covered_resources`.

## Why this matters

For a security tool, a false "all clear" is the most dangerous output there
is. This release makes coverage a first-class, visible property — which also
surfaces the cloud-agnostic breadth added in v0.4.5 and points users at the
providers they can already scan.

## Upgrade

```bash
pip install --upgrade cairn-iac   # -> 0.5.1
```

Additive and safe: existing findings and exit codes are unchanged; the only
behavioral change is that scans containing unsupported-provider resources
now say "checked N of M" instead of "clean", and JSON gains a `coverage`
block. `pip install cairn-iac==0.5.0` is a clean rollback.
